Automation is what separates OpenClaw from a smarter chatbot. With cron jobs, webhooks, and HEARTBEAT.md, your agent works for you continuously — not just when you remember to ask it something.

## Cron Jobs

Standard cron syntax for time-based scheduling. Your agent runs a prompt at the specified interval and sends the output to a channel.

```json
{
  "cron": [
    {
      "name": "morning-briefing",
      "schedule": "0 7 * * 1-5",
      "prompt": "Give me my morning briefing:\n- Today's weather\n- What's on my calendar\n- Any urgent emails I should know about\n- One thing I should focus on today",
      "channel": "imessage"
    },
    {
      "name": "weekly-review",
      "schedule": "0 17 * * 5",
      "prompt": "Prepare my weekly review:\n- What I accomplished this week\n- What carried over\n- Top priority for next week",
      "channel": "slack"
    }
  ]
}
```

**Cron expression format:** `minute hour day month weekday`

| Expression | Meaning |
|---|---|
| `0 7 * * 1-5` | 7 AM weekdays |
| `0 9 * * *` | 9 AM every day |
| `*/30 * * * *` | Every 30 minutes |
| `0 0 * * 0` | Midnight every Sunday |

### Preventing Cron Drift

For high-frequency jobs (every few minutes), use `anchorMs` to anchor the schedule to absolute timestamps rather than relative timing. This prevents small delays from accumulating into significant drift.

```json
{
  "cron": [
    {
      "name": "monitoring",
      "schedule": "*/5 * * * *",
      "anchorMs": 0,
      "prompt": "Check system status and report any anomalies",
      "channel": "slack"
    }
  ]
}
```

Daily jobs don't typically need `anchorMs`.

## Webhooks

Webhooks let external events trigger your agent. A GitHub push, a form submission, a payment received — anything that can send an HTTP request can trigger a prompt.

```json
{
  "webhooks": [
    {
      "name": "github-push",
      "path": "/webhooks/github",
      "secret": "${GITHUB_WEBHOOK_SECRET}",
      "prompt": "A new push just happened to the repo:\nBranch: {{branch}}\nCommits: {{commits}}\nAuthor: {{author}}\n\nSummarize what changed and flag anything notable.",
      "channel": "slack"
    }
  ]
}
```

Template variables (`{{branch}}`, `{{commits}}`) pull values from the incoming webhook payload.

**Security:** Always use `secret` parameters with webhooks — this verifies requests are legitimately from the expected source. Store secrets as environment variables, not inline values.

Register your webhook URL with the external service as:
```
https://your-gateway-url/webhooks/github
```

## HEARTBEAT.md

HEARTBEAT.md defines proactive behaviors that trigger during idle time. Instead of waiting for your message, the agent checks priority items, surfaces important context, and anticipates needs.

```markdown
# HEARTBEAT.md

When you have idle time between conversations:

1. Check if any active projects have unread GitHub notifications
2. Surface any calendar events in the next 2 hours that need prep
3. Flag any emails that have been waiting more than 24 hours for a response
4. If it's been more than 3 days since I logged exercise, mention it
5. Review MEMORY.md for anything time-sensitive
```

HEARTBEAT.md is what makes an agent feel like it's actually working for you rather than sitting idle waiting to be asked something.

## Session Hooks

Hooks run automatically at conversation lifecycle events:

```json
{
  "hooks": [
    {
      "type": "boot-md",
      "files": ["SOUL.md", "IDENTITY.md", "USER.md", "MEMORY.md"]
    },
    {
      "type": "session-memory",
      "schedule": "end-of-session",
      "outputPath": "~/.openclaw/logs/daily/",
      "prompt": "Summarize this conversation in 3-5 bullets.\nInclude: key decisions, tasks created, important context."
    },
    {
      "type": "command-logger",
      "outputPath": "~/.openclaw/logs/commands/",
      "format": "{{timestamp}} | {{tool}} | {{args}}"
    }
  ]
}
```

| Hook | When It Runs | Use For |
|---|---|---|
| `boot-md` | Session start | Load context files |
| `session-memory` | Session end | Save conversation summaries |
| `command-logger` | Every tool call | Audit trail |

## Channel Strategy for Automation

Different automations belong on different channels to avoid notification fatigue:

- **iMessage:** Urgent, time-sensitive, personal (morning briefing, immediate alerts)
- **Slack `#monitoring` channel:** Automated monitoring output, system status
- **Slack `#assistant` channel:** Work-related briefings and updates
- **WebChat:** Testing and debugging automations

If everything goes to one channel, important alerts get buried in routine output.

## Cost Monitoring

Recurring automations consume API tokens continuously. A daily briefing typically costs $1.50–$4.50 per month depending on the model and response length. Hourly monitoring adds up quickly.

Check cron status:
```bash
openclaw cron status
```

Other useful cron management commands:
```bash
openclaw cron list        # List all scheduled jobs
openclaw cron add         # Add a new cron job
openclaw cron edit        # Edit an existing cron job
```

Track costs by automation:
```bash
openclaw costs --breakdown cron
```

Consider cheaper models for routine automations and reserve your primary model for interactive conversations.
