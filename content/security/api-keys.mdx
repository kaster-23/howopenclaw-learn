<Callout type="error" title="Never store credentials in SOUL.md or config files">
These files sync across devices, get backed up to cloud storage, and regularly appear in community help threads. A leaked API key means unauthorized charges, data access, and account compromise. Use environment variables or the built-in secrets manager instead.
</Callout>

The most prevalent security error in OpenClaw setups: embedding API credentials directly in SOUL.md or other configuration files.

These files sync across devices, get backed up to cloud storage, show up in community help threads, and are easy to accidentally share. Your API keys should never live in a markdown file.

## The Three Safe Approaches

### 1. Environment Variables

Store credentials in your shell profile, reference them by variable name in configuration files.

```bash
# ~/.zshrc or ~/.bashrc
export ANTHROPIC_API_KEY="sk-ant-..."
export OPENAI_API_KEY="sk-..."
export SLACK_BOT_TOKEN="xoxb-..."
```

Reference in config:
```json
{
  "providers": {
    "anthropic": {
      "apiKey": "${ANTHROPIC_API_KEY}"
    }
  }
}
```

The `${VAR_NAME}` syntax is supported throughout `openclaw.json` and channel configuration files.

### 2. OpenClaw Secrets Management

The built-in secrets manager encrypts credentials on disk and never exposes them in logs or config files.

```bash
openclaw secrets set ANTHROPIC_API_KEY sk-ant-...
openclaw secrets set SLACK_BOT_TOKEN xoxb-...
openclaw secrets list  # Shows key names only, never values
```

Reference in config the same way as environment variables: `${ANTHROPIC_API_KEY}`.

### 3. OAuth (Preferred When Available)

For services that support it, OAuth is better than API keys:
- **Scoped** — limited to exactly the permissions you granted
- **Time-limited** — tokens expire and rotate automatically
- **Revocable** — cancel access instantly without rotating keys
- **Auditable** — provider logs show what your app accessed

Use OAuth for Google (Calendar, Gmail), GitHub, and any service that offers it.

## Tool Risk Classification

Not all tool permissions are equal. Classify and configure accordingly:

### Low Risk — Enable Freely
- File reading (allowed paths only)
- Web search and browsing
- Weather fetching
- Calendar viewing (read-only)

### Medium Risk — Enable With Logging
- Web browsing with page actions
- Messaging (your own channels)
- File writing (workspace only)
- Calendar modifications

### High Risk — Require Approval
- Shell command execution
- Package installation
- Email sending
- Arbitrary HTTP requests

## The `requireApproval` Pattern

For high-risk tools, enable human-in-the-loop oversight:

```json
{
  "tools": {
    "shellExec": {
      "enabled": true,
      "requireApproval": true,
      "approvalChannel": "imessage"
    },
    "emailSend": {
      "enabled": true,
      "requireApproval": true
    }
  }
}
```

When `requireApproval: true`, the agent sends you a confirmation request before executing the tool. You approve or deny. This lets you build confidence in what the agent does without fully disabling capabilities.

## Allowlists and Blocklists

Constrain high-risk tools to specific commands or domains:

```json
{
  "tools": {
    "shellExec": {
      "commandAllowlist": ["ls", "cat", "grep", "git status", "git log"],
      "commandBlocklist": ["rm -rf", "sudo", "curl | sh"]
    },
    "httpRequest": {
      "domainAllowlist": ["api.anthropic.com", "api.openai.com", "api.slack.com"],
      "blockPrivateNetworks": true
    }
  }
}
```

## Signs Your Keys Have Been Compromised

- Unexpected API usage spikes
- Charges from providers you didn't use
- Unfamiliar API requests in provider logs
- OpenClaw logs showing tool calls you didn't trigger

If you suspect compromise: immediately revoke all keys, generate new ones, audit your installed skills, and check your SOUL.md and any shared config files for embedded credentials.
