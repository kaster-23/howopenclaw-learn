OpenClaw runs locally by default. That's fine for testing, but if you want a truly always-on agent, you'll want to run it on a machine that's never sleeping. Here are your options.

## Option 1: Local Deployment

Your laptop or desktop, always-on.

**Pros:** Zero additional cost, simplest setup.
**Cons:** Stops when your computer sleeps. No redundancy.

**When to use:** Learning and testing. Not recommended for production.

If you go this route, configure your Mac or Linux machine to never sleep:

```bash
# macOS — disable sleep
sudo pmset -a sleep 0

# Or install OpenClaw as a launch agent
openclaw gateway install
```

---

## Option 2: Docker Deployment

Containerized OpenClaw for consistent, reproducible environments.

**Pros:** Works identically on any machine. Easy to move and restore. Good isolation.
**Cons:** More complex setup. Still needs a machine that's always on.

```dockerfile
FROM node:24-alpine

WORKDIR /app
RUN npm install -g openclaw@latest

VOLUME ["/root/.openclaw"]

EXPOSE 18789 3000

CMD ["openclaw", "gateway"]
```

```bash
# Build and run
docker build -t my-openclaw .
docker run -d \
  --name openclaw \
  --restart unless-stopped \
  -v ~/.openclaw:/root/.openclaw \
  -p 127.0.0.1:18789:18789 \
  my-openclaw
```

Docker Compose makes management easier:

```yaml
version: "3.8"
services:
  openclaw:
    build: .
    restart: unless-stopped
    volumes:
      - ~/.openclaw:/root/.openclaw
    ports:
      - "127.0.0.1:18789:18789"
    env_file: .env
```

## Option 3: VPS Deployment (Recommended for Always-On)

A small VPS runs 24/7 and costs $4–12/month. This is what most serious users end up choosing.

**Recommended providers:**

| Provider | Cheapest Plan | Notes |
|---|---|---|
| Hetzner | ~$4/month | Best price/performance in Europe |
| Fly.io | ~$5/month | Easy deploy from CLI |
| GCP Compute Engine | ~$6/month | Google Cloud, good global reach |
| Azure Linux VM | ~$5/month | Microsoft Cloud option |
| exe.dev | ~$5/month | Developer-focused hosting |

**Setup on Ubuntu 22.04:**

```bash
# Install Node.js 24
curl -fsSL https://deb.nodesource.com/setup_24.x | sudo -E bash -
sudo apt-get install -y nodejs

# Install OpenClaw
npm install -g openclaw@latest

# Configure
openclaw onboard --install-daemon

# Run as systemd service
openclaw gateway install
openclaw gateway start
systemctl --user enable openclaw
systemctl --user start openclaw
```

**Security basics for a VPS:**
- Disable password auth, use SSH keys only
- Configure UFW to allow only ports you need
- Keep the gateway bound to `127.0.0.1` (not `0.0.0.0`)
- Use Tailscale for remote access (see Option 4)

## Option 4: Tailscale + VPS (Best Security)

Tailscale creates an encrypted private network between your devices. Your VPS runs OpenClaw, but the gateway is only accessible through the Tailscale network — never exposed to the public internet.

```bash
# On your VPS
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up

# Get your VPS's Tailscale IP
tailscale ip -4
```

Now access your agent from any device on your Tailscale network:
```
http://your-vps-tailscale-ip:18789
```

**Cost:** Free for personal use (up to 100 devices).

**Security tier:** Much lower risk than a public VPS — gateway never touches the internet.

## Recommended Progression

Most users follow this path over 3+ months:

1. **Week 1–2:** Local deployment — learn the system
2. **Week 3–4:** Docker on local machine — practice the container workflow
3. **Month 2+:** VPS with Tailscale — production-ready, always-on

Don't jump to VPS on day one. There's a learning curve, and debugging remotely is harder than debugging locally.

## Backup Your Configuration

Before you deploy anywhere, set up automated backups.

```bash
# Simple backup script
#!/bin/bash
DATE=$(date +%Y%m%d)
tar -czf ~/openclaw-backup-$DATE.tar.gz \
  ~/.openclaw/SOUL.md \
  ~/.openclaw/USER.md \
  ~/.openclaw/MEMORY.md \
  ~/.openclaw/openclaw.json \
  ~/.openclaw/channels/ \
  ~/.openclaw/workspace/

# Upload to your preferred backup destination
```

Schedule it with cron:
```bash
0 2 * * * /path/to/backup.sh
```

Your config is recoverable. Your MEMORY.md and SOUL.md are not easily replaced — back them up.
