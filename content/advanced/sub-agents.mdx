Sub-agents are background agents your main agent spawns to handle specific tasks independently. While the sub-agent works, your main agent can continue processing other things — or wait for results and synthesize them.

## What Is a Sub-Agent?

> A sub-agent is a background agent that your main agent spawns to handle a specific task independently.

Each sub-agent gets:
- A specific instruction (its task)
- Its own context (isolated from the main conversation)
- Access to specified skills
- A timeout limit
- A callback when complete

## Creating Sub-Agents

The `sessions_spawn` command creates a sub-agent:

```javascript
const result = await sessions_spawn({
  instruction: "Research the top 5 competitors of Acme Corp. Focus on pricing, key features, and recent news.",
  model: "claude-haiku-4-5",     // Cheaper model for research tasks
  skills: ["web-search", "web-fetch"],
  timeout: 120000,               // 2 minutes
  onComplete: "summarize_research"
})
```

For parallel work, spawn multiple simultaneously:

```javascript
// Parallel morning briefing components
const [weather, calendar, emails, news] = await Promise.all([
  sessions_spawn({ instruction: "Get today's weather for my location", skills: ["weather"] }),
  sessions_spawn({ instruction: "Summarize my calendar for today", skills: ["google-calendar"] }),
  sessions_spawn({ instruction: "Triage urgent emails", skills: ["gmail"] }),
  sessions_spawn({ instruction: "Top 3 tech news items relevant to my work", skills: ["web-search"] }),
])

// Synthesize results
return synthesize(weather, calendar, emails, news)
```

## When to Use Sub-Agents

**Use sub-agents when:**
- The task will take more than 30 seconds
- Multiple independent pieces of work can run simultaneously
- A task requires specialized skills you don't want loaded in the main context
- You want to protect the main conversation's context window from large research outputs

**Don't use sub-agents for:**
- Simple, quick tasks (overhead isn't worth it)
- Deeply interdependent steps where each depends on the last
- Tasks requiring back-and-forth conversation

## Architecture Patterns

### Coordinator + Workers

One manager distributes tasks to specialist sub-agents:

```
Main Agent (Coordinator)
├── Research Sub-Agent (web search, news)
├── Data Sub-Agent (spreadsheet, calculations)
└── Writing Sub-Agent (draft generation)
```

### Pipeline

Sequential stages where output feeds forward:

```
Main Agent → Research Agent → Outline Agent → Draft Agent
```

Each stage hands off to the next rather than running in parallel.

### Fan-Out / Fan-In

Multiple agents gather data, main agent synthesizes:

```
Main Agent
├── Competitor A Research
├── Competitor B Research     → All results → Main synthesizes
├── Competitor C Research
└── Competitor D Research
```

### Specialist Routing

Main agent acts as dispatcher to domain-specific sub-agents:

```
User Request → Main Agent (dispatcher)
├── If code question → Dev Sub-Agent
├── If financial → Finance Sub-Agent
└── If writing → Content Sub-Agent
```

## Model Selection for Sub-Agents

Sub-agents don't need your most capable model. Match the model to the task:

| Task | Suggested Model | Why |
|---|---|---|
| Web research | Claude Haiku or GPT-4o-mini | Fast, cheap, sufficient |
| Data processing | Claude Haiku | Structured tasks, no reasoning needed |
| Complex analysis | Claude Sonnet | Needs reasoning |
| Final synthesis | Your primary model | Quality matters here |

Using cheaper models for sub-agents can reduce costs by 10-50x compared to using your primary model for everything.

## Key Limitations

- **No sub-agent spawning.** Sub-agents cannot spawn their own sub-agents. The hierarchy is flat: main agent + workers only.
- **Context isolation.** Sub-agents don't inherit your conversation history. Give them all the context they need in the `instruction` field.
- **Practical concurrency limit.** 3–5 simultaneous sub-agents is the practical maximum before API rate limits become a problem.
- **No direct communication.** Sub-agents can't talk to each other. All coordination flows through the main agent.

## Practical Example: Research Report

```javascript
// Spawn parallel research sub-agents
const [market, competitors, tech, regulatory] = await Promise.all([
  sessions_spawn({
    instruction: "Research the current market size and growth trends for B2B SaaS in healthcare",
    skills: ["web-search", "web-fetch"],
    model: "claude-haiku-4-5",
  }),
  sessions_spawn({
    instruction: "Find the top 5 competitors in healthcare SaaS, their pricing and key features",
    skills: ["web-search", "web-fetch"],
    model: "claude-haiku-4-5",
  }),
  sessions_spawn({
    instruction: "What are the key technical trends in healthcare SaaS for 2026?",
    skills: ["web-search"],
    model: "claude-haiku-4-5",
  }),
  sessions_spawn({
    instruction: "What regulatory changes (HIPAA, etc.) are affecting healthcare SaaS in 2026?",
    skills: ["web-search"],
    model: "claude-haiku-4-5",
  }),
])

// Synthesize with primary model
return synthesizeReport(market, competitors, tech, regulatory)
```

Four parallel research tasks complete in the time it would take to run one sequentially.
