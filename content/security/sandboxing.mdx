---
title: "Sandboxing"
description: "Restrict what your OpenClaw agent can access on your system to limit the blast radius of prompt injection, buggy skills, or misconfiguration."
---

Sandboxing restricts your OpenClaw agent's capabilities on the host system. It's your protection against prompt injection attacks, buggy skills, and misconfigured tools doing damage they can't undo.

## Why Sandboxing Matters

<Callout type="warn" title="Sandboxing is not enabled by default">
A fresh OpenClaw install has no file system restrictions. The agent can read any file and execute any shell command available to your user account. Configure sandboxing before installing community skills or granting tool access.
</Callout>

A fully unrestricted agent can read any file, execute any command, and access any path on your system. That's powerful — and dangerous. Sandboxing creates explicit boundaries around what the agent is allowed to touch.

The goal isn't to cripple your agent. It's to contain the blast radius if something goes wrong.

## Built-in Sandbox Mode

The simplest option: enable `sandbox: true` in your configuration.

```json
{
  "sandbox": {
    "enabled": true,
    "fileSystem": {
      "allowedPaths": ["~/.openclaw/workspace/", "~/.openclaw/workspace/"],
      "blockedPaths": ["~/.ssh", "~/.aws", "~/.config", "~/.gnupg", "~/.kube"],
      "readOnly": ["~/.openclaw/SOUL.md", "~/.openclaw/USER.md"]
    },
    "commands": {
      "allowlist": ["ls", "cat", "grep", "find", "echo"],
      "blockShell": false
    },
    "network": {
      "restrictOutbound": false
    }
  }
}
```

What this enables:
- File system restrictions to specific allowed directories
- Command filtering through allowlists
- Optional network restrictions
- Blocks system-level operations

## Critical Directories to Protect

Always deny agent access to these locations:

| Directory | Why |
|---|---|
| `~/.ssh` | SSH keys — full server access if exfiltrated |
| `~/.aws` | AWS credentials |
| `~/.config` | Application tokens and configs |
| `~/.gnupg` | GPG keys |
| `~/.kube` | Kubernetes cluster access |
| `/etc` | System configuration files |

A good rule of thumb: block any hidden directory (`.*`) that isn't explicitly part of OpenClaw's workspace.

## Docker Isolation (Advanced)

For the highest level of isolation, run OpenClaw inside a Docker container. This is the recommended approach if you're running it on a shared system or exposing it remotely.

```dockerfile
FROM node:24-alpine

WORKDIR /app

COPY . .
RUN npm install -g openclaw@latest

# Mount configs read-only
VOLUME ["/root/.openclaw:ro"]
# Dedicated workspace volume
VOLUME ["/root/.openclaw/workspace"]

EXPOSE 18789

CMD ["openclaw", "start"]
```

Docker isolation features:
- Read-only config mounts — agent can read but not modify
- Dedicated workspace volumes separate from your host filesystem
- Localhost-only port binding by default
- Resource caps (memory and CPU limits)

```bash
docker run \
  --memory="512m" \
  --cpus="0.5" \
  -v ~/.openclaw:/root/.openclaw:ro \
  -v ~/.openclaw/workspace:/root/.openclaw/workspace \
  -p 127.0.0.1:18789:18789 \
  openclaw
```

## What Level of Sandboxing Do You Need?

| Use Case | Recommendation |
|---|---|
| Personal local setup, trusted skills only | Built-in sandbox mode |
| Multiple ClawHub skills installed | Built-in sandbox + blocked sensitive paths |
| Running on a VPS | Docker isolation |
| Shared system or team deployment | Docker + network restrictions |

## Sandboxing and Skill Permissions

Skills request permissions at installation time. If a skill requests access to paths outside your workspace, that's a red flag. Deny the permission and investigate the skill before enabling it.

With sandbox mode enabled, skills that try to access blocked paths will receive an error rather than silently succeeding — which makes it much easier to identify misbehaving skills.
