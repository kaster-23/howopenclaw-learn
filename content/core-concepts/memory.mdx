---

OpenClaw's memory system transforms the agent from a stateless chatbot into a persistent assistant that actually knows you over time. It works in three layers, each serving a different purpose.

## The Three Memory Layers

### Short-Term Memory

The current conversation context — everything said in this session. Automatic. Lives in the context window. Disappears when the session ends.

You don't configure short-term memory. It just works.

### Medium-Term Memory

Daily logs automatically generated by the `session-memory` hook. Each conversation summary gets appended with a timestamp and topic label.

```json
{
  "hooks": [
    {
      "type": "session-memory",
      "schedule": "end-of-session",
      "outputPath": "~/.openclaw/logs/daily/",
      "format": "{{date}} | {{topics}} | {{summary}}"
    }
  ]
}
```

These logs accumulate over days and weeks, giving the agent access to what you discussed recently without overwhelming the context window with everything from the past year.

### Long-Term Memory

`MEMORY.md` — a manually curated file of persistent facts, active projects, preferences, and discovered patterns. This is what the agent loads at startup.

```markdown
# MEMORY.md

## Active Projects
- Launching new product line Q2 2026
- Learning Spanish — intermediate level

## Preferences
- Prefers bullet points over numbered lists
- Responds to messages in batches, not immediately
- Works from 6-11am, 2-5pm

## Key Context
- Main collaborator: Sarah (engineering lead)
- Current challenge: scaling the data pipeline
```

Keep MEMORY.md under 200 lines. Quality over quantity.

## How Memory Search Works

When you ask your agent something that might benefit from past context, it uses a **hybrid search** combining:

- **70% semantic search** — vector embeddings for conceptual matching ("what did we discuss about the deployment issues?")
- **30% BM25 keyword matching** — exact term recall ("show me the notes from March 15th")

This combination means both "remind me what we talked about regarding scaling" and "find the Kubernetes config we discussed" will return relevant results.

## The Boot-MD Hook

At conversation start, the `boot-md` hook automatically loads your identity files into context:

```json
{
  "hooks": [
    {
      "type": "boot-md",
      "files": [
        "SOUL.md",
        "IDENTITY.md",
        "USER.md",
        "MEMORY.md"
      ]
    }
  ]
}
```

Everything in these files is available from the first message of every conversation. This is why MEMORY.md file size matters — it's loaded into the context window every time.

## Memory Maintenance

> Stale memory is worse than no memory.

An outdated MEMORY.md entry about a completed project, a resolved issue, or a person you no longer work with actively misleads your agent. It's worse than not having the entry at all.

**Weekly review:** Spend 5 minutes reviewing recent daily logs. Promote important patterns into MEMORY.md. Remove entries that are no longer current.

**Monthly compaction:** Consolidate multiple related entries into single, more general truths. "Discussed deployment options 4 times" → "Prefers Docker over bare metal for new projects."

**Quarterly purge:** Remove everything that's no longer true or relevant. Projects end. People change roles. Keep MEMORY.md current.

## Writing Good Memory Entries

**Be specific:**
❌ "Works on technical stuff"
✅ "Senior engineer at Acme Corp, focused on backend infrastructure and Kubernetes"

**Be actionable:**
❌ "Likes coffee"
✅ "Best time to discuss complex topics: morning, before 10am"

**Be current:**
❌ "Working on Project Phoenix" (if it launched 6 months ago)
✅ "Launched Project Phoenix in October 2025; currently in maintenance mode"

## Memory vs. Context Window

Don't try to put everything in MEMORY.md and load it all into every conversation. You'll hit context limits and slow down every interaction.

The right pattern:
- **MEMORY.md:** Persistent facts that apply to most conversations
- **Daily logs:** Recent context retrieved on demand via search
- **Session context:** Everything from the current conversation

The hybrid search means your agent can retrieve specific past context without loading all of it upfront.
